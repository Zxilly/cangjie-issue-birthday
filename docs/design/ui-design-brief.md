# Cangjie Issue 生日蛋糕机 — 界面设计需求文档

> 交付对象：UI/UX 设计师
> 版本：v1 · 2026-06-15
> 文档目的：让设计师在不接触代码的情况下，完整理解产品、用户、各页面与状态、交互动效、文案调性、响应式与可访问性要求，并明确需要交付的设计产物。

---

## 1. 一句话定位

一个轻量的网页小工具：开发者用 GitCode 账号登录后，能看到**自己在 Cangjie 组织里、仍未关闭、且已经"满 1 岁"的老 issue**，给每条标注"几周岁"，并能**一键送上一个生日蛋糕**（在该 issue 下发一条庆祝评论）。

整体气质：**温暖、轻松、有仪式感的"过生日"主题**，同时不失开发者工具应有的清爽与可信。不要做成幼稚的卡通玩具，目标是"开发者会心一笑并愿意点一下"的精致小品。

---

## 2. 目标用户与使用场景

- **用户**：在 GitCode 的 Cangjie（仓颉编程语言）组织里提过 issue 的开发者；熟悉 GitHub/GitCode 类工具，审美偏向简洁现代。
- **典型场景**：用户打开网页 → 用 GitCode 登录 → 看到一串自己很久以前提、至今没关掉的 issue → 觉得有趣 → 给其中几个点"送蛋糕"，给这些"长寿 issue"一点幽默的仪式感。
- **使用时长**：一次几十秒到几分钟的轻量互动，多为偶发性把玩，而非高频生产力工具。
- **设备**：桌面浏览器为主，移动端浏览器需可用（响应式）。

---

## 3. 设计目标与原则

1. **第一眼读懂**：未登录用户 3 秒内明白"这是干嘛的、为什么要登录"。
2. **年龄信息一眼可见**：每个 issue 的"周岁"是卡片的视觉焦点之一。
3. **送蛋糕要有爽感**：一键操作，配一个短促的庆祝动效，让人想多点几个。
4. **状态清晰**：按钮在"可送 / 检测中 / 发送中 / 已送 / 失败"之间的视觉区分明确，绝不让用户误以为重复发了。
5. **克制的趣味**：生日主题元素（蛋糕、蜡烛、彩带、气球）点到为止，服务于信息，不喧宾夺主。

---

## 4. 整体调性与风格方向

- **情绪关键词**：庆祝、温暖、俏皮、干净、可信。
- **风格方向（供设计师发挥，非锁定）**：现代扁平 + 柔和圆角 + 适度留白；可用一种"生日感"的主题色点缀（如暖橙/玫瑰粉/奶油黄之一作强调色），主色调保持中性、不刺眼。
- **品牌关系**：本工具是面向 Cangjie 社区的非官方趣味工具，**可以呼应仓颉/GitCode 的视觉气质但不必照搬**；是否使用仓颉品牌色由设计师提案、产品方确认（见第 14 节）。
- **避免**：过度拟物的卡通蛋糕大图、廉价 emoji 堆砌、高饱和撞色、模板感强的 Bootstrap 既视感。

---

## 5. 信息架构与页面清单

产品很小，只有一条主线 + 若干状态。需要设计的"屏/态"如下：

| 编号 | 名称 | 说明 |
|---|---|---|
| S1 | 登录前落地页 | 未登录时的首屏，含产品说明 + 登录入口 |
| S2 | 登录跳转/回调过渡态 | 点击登录后、以及从 GitCode 跳回时的加载过渡 |
| S3 | 主列表页 | 登录后的核心页：顶栏 + issue 列表 |
| S3-a | 列表加载态 | 拉取 issue 时的骨架屏/加载 |
| S3-b | 空状态 | 没有符合条件的老 issue 时 |
| S3-c | 错误态 | 拉取失败 / 登录失效 / 触发限流 |
| C1 | Issue 卡片（多态） | 列表中的单元，含蛋糕按钮的全部状态 |
| C2 | 全局通知 / Toast | 送蛋糕成功、失败、限流等反馈 |
| C3 | 送蛋糕庆祝动效 | 成功瞬间的微动效（如彩纸） |

> 没有多级路由、没有设置页、没有详情页（点 issue 标题直接外链跳到 GitCode 原页）。请勿扩展额外页面，保持极简。

---

## 6. 各页面 / 状态详细需求

### S1 · 登录前落地页

**目标**：说清楚是什么 + 一键登录。

需包含：
- 产品名 / 标识（文字或简单图形 logo，设计师可提案一个轻量 logo）。
- 一句话副标题，传达"给你在 Cangjie 的老 issue 过生日"。
- 简短说明（1–2 句）：登录后会展示你创建的、超过 1 年还没关闭的 issue。
- **主行动按钮**："用 GitCode 登录"（带 GitCode 标识/图标）。
- 一行很小的安全/隐私说明位（文案见第 9 节）。

验收标准：新用户无需滚动即可看懂用途并找到登录按钮；登录按钮是页面唯一的主 CTA。

### S2 · 登录跳转 / 回调过渡态

- 点击登录后到跳转 GitCode 之间，以及从 GitCode 带 `code` 跳回本站、前端在后台换取令牌时，会有**短暂等待**。
- 需要一个**轻量全屏/居中加载态**：一个生日主题的加载指示（如旋转的蜡烛/蛋糕，或简单 spinner + 文案"正在登录…"）。
- 该态可能持续 1–3 秒，请确保不显得卡死。

### S3 · 主列表页（核心）

布局自上而下：

1. **顶栏（Header，见 C-Header）**：左侧产品名/logo；右侧当前登录用户头像 + 昵称 + "退出"。
2. **列表概要**：一行汇总文案，如"你有 **N** 个超过 1 岁的老 issue 🎂"。如果想做轻量统计（如最年长的岁数）可提案。
3. **Issue 列表**：纵向排列的 Issue 卡片（C1）。默认排序：**最年长（创建最早）在最前**。是否提供排序/筛选控件（如按年龄/按仓库）由设计师提案，非必需。
4. （可选）页脚：作者/项目来源链接，弱化处理。

验收标准：在常见桌面宽度下，单屏能看到概要 + 至少 2–3 张完整卡片；信息层级清晰，"周岁"与"送蛋糕"是每张卡的两个视觉重点。

### S3-a · 列表加载态

- 拉取期间显示**骨架屏**（卡片轮廓的占位），而非整屏 spinner，减少跳动。
- 数据可能需要分页拉取，过程稍长，骨架屏要能自然承接。

### S3-b · 空状态

- 当用户没有任何"满 1 岁且未关闭"的 issue 时显示。
- 需要一个**友好、略带幽默**的空状态插画/图形 + 文案（文案见第 9 节），传达"这是好事（你没有积压的老 issue）"，而非错误感。

### S3-c · 错误态

需要覆盖三类，可共用一套错误展示组件，文案不同：
- **登录失效**：提示重新登录，给"重新登录"按钮。
- **加载失败**（网络/接口错误）：提示 + "重试"按钮。
- **触发限流**（请求过于频繁）：提示稍后再试，建议给一个"稍后重试"的引导（文案见第 9 节）。

## 7. 核心组件规格

### C-Header · 顶栏

- 元素：产品名/logo、用户头像（圆形）、昵称、退出入口。
- 移动端可折叠为头像 + 菜单。
- 退出为次级操作，弱化但可达。

### C1 · Issue 卡片（重点组件，需出全部状态）

单张卡片需展示的信息：

| 字段 | 示例 | 设计要点 |
|---|---|---|
| Issue 标题 | "请清理一下标准库里的 XXX" | 主信息，最高字重；过长截断（建议两行省略号）；点击外链跳 GitCode 原页 |
| 所属仓库 | `cangjie_runtime` | 次要标签样式（如 pill/灰底） |
| 创建日期 | 2023-04-20 | 弱化展示 |
| **周岁徽章** | "3 周岁" | **视觉焦点之一**，见下方"周岁徽章" |
| 外链入口 | ↗ | 跳转 GitCode 原 issue（新标签页） |
| **送蛋糕按钮** | 🎂 送蛋糕 | **视觉焦点之二**，见 C1-按钮状态 |

**周岁徽章设计要求**：
- 必须让"几周岁"一眼可读，是卡片的情绪亮点。
- 可考虑随年龄增强视觉强调（例如越老越"隆重"：颜色更暖、加蜡烛数等），由设计师提案。给出至少"1 周岁 / 3 周岁 / 5 周岁以上"几档示意。
- 文案统一为"N 周岁"。

**C1-按钮状态机（必须全部出图）**：

| 状态 | 触发 | 视觉要求 |
|---|---|---|
| 可送 | 默认 | 主按钮样式，明确可点，含 🎂 与"送蛋糕"文案 |
| 检测中 | 点击后先检查是否已送过 | 按钮内 loading，禁用，文案如"检测中…" |
| 发送中 | 正在提交评论 | 按钮内 loading，禁用，文案如"发送中…" |
| 已送 | 本次发送成功 / 检测到此前已送 | 置为"已送 🎂"灰态/成功态，**明确不可再点**，避免误以为没成功 |
| 失败 | 提交失败 | 错误态 + "重试"，不破坏卡片布局 |

> "已送"状态非常重要：用户绝不能因为不确定是否成功而重复点击。请让成功/已送状态在视觉上不可误解。

### C2 · 全局通知 / Toast

- 用于送蛋糕成功、失败、限流等即时反馈。
- 轻量、自动消失、不遮挡操作；成功类可带一点庆祝色彩。

### C3 · 送蛋糕庆祝动效

- 送出成功的瞬间，给一个**短促、克制的庆祝动效**（如彩纸 confetti、蛋糕"点亮蜡烛"微动画）。
- 时长建议 < 1.5 秒，不阻塞继续操作。
- **必须提供"减少动效"降级方案**（见第 11 节）。

---

## 8. 关键交互与动效

- **一键送蛋糕**：点击 → 自动检测是否已送 → 未送则提交 → 成功庆祝动效 + 按钮转"已送"。整个过程无需二次确认弹窗（设计为"一键直发"）。如设计师认为需要轻量确认/预览，可提案一个**不打断爽感**的方案（如气泡预览即将发送的祝福语）。
- **蛋糕祝福语**：发送内容是**固定趣味模板**（自动带入周岁），用户不编辑。若设计中需要展示"将要发送的内容"，可参考第 9 节的示例文案做预览。
- **悬停/反馈**：卡片与按钮需有清晰的 hover/active/focus 反馈。
- 动效整体追求"轻快"，避免冗长过场。

---

## 9. 文案与语气（Copywriting）

- **界面语言**：简体中文。
- **语气**：温暖、俏皮、口语化，但不油腻；偶尔用 1 个 emoji 提味，避免堆砌。

需要设计师配合定稿/排版的关键文案（以下为参考草案，可优化）：

- 落地页副标题：例 "给你在 Cangjie 的老 issue，补过一个生日 🎂"
- 登录按钮："用 GitCode 登录"
- 安全提示（小字）："仅用于读取你在 Cangjie 的 issue 并代你发送祝福评论。"
- 列表概要："你有 N 个超过 1 岁的老 issue，要不要给它们过个生日？"
- 空状态："太棒了，你在 Cangjie 没有满 1 岁的老 issue 🎉"
- 限流提示："手速太快啦，喝口水稍等一下再送 🍰"
- 蛋糕祝福语示例（实际为固定模板，含周岁与蛋糕图样，发布到 issue 评论区）：
  > 🎂 这个 issue 已经 **3 周岁** 啦！祝它生日快乐，也愿它早日被关闭～🎉

---

## 10. 响应式与适配

- **桌面优先**，但需给出**移动端（窄屏）**布局：卡片单列、顶栏折叠、按钮触达区域足够大（≥ 44px 命中区）。
- 建议给出至少两个断点的稿：桌面（约 ≥1024px）与移动（约 ≤480px）；中间尺寸自适应。
- 列表在超宽屏下需有最大内容宽度，避免一行过长。

---

## 11. 可访问性（必须满足）

- **对比度**：正文与背景对比度达到 WCAG AA（正常文本 ≥ 4.5:1）。
- **键盘可达**：登录、送蛋糕、退出、外链等均可 Tab 聚焦并有清晰焦点样式。
- **状态不只靠颜色**：按钮"已送/失败"等状态需同时有图标或文字区分，不能仅靠红绿色。
- **动效降级**：尊重系统"减少动效"偏好（prefers-reduced-motion），此时关闭/弱化庆祝动效与过场。
- **可读字号**：正文不小于 14px；周岁徽章等关键信息更大。

---

## 12. 视觉规范需求（期望设计师定义并交付）

请在设计中明确并沉淀为可复用规范：

- **色彩**：主色、强调/生日主题色、中性灰阶、语义色（成功/失败/警示）。是否做**暗色模式**：建议提供，但可作为加分项（与产品方确认优先级）。
- **字体**：中文与西文字族、字号阶梯、字重。
- **间距与栅格**：基础间距单位、卡片内外边距、列表间距。
- **圆角 / 阴影 / 描边**：统一的圆角档位与层级阴影。
- **图标**：所需图标清单（GitCode 标识、外链、退出、蛋糕/蜡烛/彩带、加载、成功/失败等）。
- **插画/图形**：空状态插画、加载指示、庆祝动效素材。

## 13. 技术约束（影响设计可行性，请知悉）

- 实现为**纯前端网页**（React + Tailwind CSS v4），无自定义后端。
- 因此：所有交互都在浏览器内完成；庆祝动效、骨架屏等需为前端可实现的轻量方案（CSS/SVG/Lottie 皆可，但请控制体积）。
- 数据来自 GitCode 接口，**列表长度不定**（可能十几条到上百条），列表设计需考虑较多条目时的滚动与性能（卡片样式不宜过重）。
- 不依赖会员/付费视觉资产；图标与插画建议使用可商用/开源或原创素材。

## 15. 假设与待确认（请产品方红线确认）

以下是我为推进而做的默认假设，若与你预期不符请直接修改：

1. **界面语言为简体中文**（面向仓颉中文社区）。是否需要中英双语？默认：仅中文。
2. **品牌色**：是否要求呼应仓颉/GitCode 官方色，还是由设计师自由提案一套"生日主题"配色？默认：设计师自由提案，可参考但不强制官方色。
3. **暗色模式**：默认建议提供但列为加分项。是否必做？
4. **logo**：是否需要设计师设计一个轻量产品 logo/标识？默认：需要一个简单标识。
5. **送蛋糕是否需要二次确认/预览**：默认"一键直发、无弹窗"，是否要加轻量预览？
6. **排序/筛选控件**：默认仅"最年长在前"，是否需要按仓库/年龄筛选？
7. **范围口径（不影响视觉，仅供设计师理解数据）**：列表只含"当前登录用户**创建的**、**未关闭的**、创建至今**超过 1 年**的 issue"。
